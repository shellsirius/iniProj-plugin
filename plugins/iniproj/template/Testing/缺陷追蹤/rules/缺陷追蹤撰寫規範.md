# 缺陷追蹤撰寫規範

規範 `Testing/缺陷追蹤/` 的缺陷（Issue）文件。測試執行後，將不符預期的問題記錄為 Issue，回扣對應測試案例（TC）。

> 與議題追蹤系統（GitLab／Jira／Redmine 等）對應：本資料夾為**人看的鏡像／草稿**，欄位通用、不綁特定平台。

---

## 一、檔名規則

```
Testing/缺陷追蹤/[狀態]#{追蹤號}_{TC編號}_{標題}.md
```

- **追蹤號**：議題系統的編號（如 #231）；尚未建立可先用流水暫號。
- **TC 編號**：對應的測試案例；多筆以底線串接（`TC011_TC012`）；跨案例／共通問題用 `共用`。
- **狀態前綴**：

| 前綴 | 意義 |
|---|---|
| （無前綴） | 待複測／處理中 |
| `[複測通過]` | 複測通過 |
| `[複測退回]` | 複測有退回項 |
| `[複測暫緩]` | 被其他 Issue 阻擋 |
| `[已關閉]` | 議題已關閉 |

> 狀態變更時改檔名前綴，內容不刪。

---

## 二、新建 Issue 結構

| 區段 | 內容 |
|---|---|
| 問題描述 | 一兩句話簡述現象 |
| 重現步驟 | 條列可重現的操作 |
| 實際結果 | 目前實際發生的狀況 |
| 預計結果 | 正確／應有的行為（可回扣 SRS／PRD 驗收條件） |
| 測試環境 | 版本／環境／帳號·角色／瀏覽器·裝置 |
| 系統截圖 | 截圖路徑（逐行標註用途） |

---

## 三、複測 Issue 結構

複測結果**不覆蓋原描述**，於原內容下方以 `---` 分隔線接續：

| 區段 | 內容 |
|---|---|
| 測試環境 | 沿用當輪設定，帳號／角色依實況 |
| 測試項目結果【通過】 | 條列通過項目，附「定稿值 vs 實測值」佐證 |
| 測試項目結果【退回】 | 退回原因＋重現步驟；無則註「（本次複測無退回項目）」 |
| 系統測試截圖 | 複測佐證截圖路徑 |

---

## 四、撰寫慣例

- **語系**：繁體中文。
- **回扣追溯**：檔名 TC 編號 → 測試案例「需求編號」→ PRD R檔，形成 R ↔ TC ↔ Issue 鏈。
- **不覆蓋**：複測接在原文之後，保留歷程。
- **敏感資訊**：實際帳密／個資不入庫，以代稱表示。
