# iniRFPtoPRD

把 `iniRFP.md`（規格初稿）轉成結構化 PRD 需求檔。

## 第 0 步：先讀規範
讀 `PRD/rules/PRD撰寫規範.md`，全程依其格式撰寫。三條鐵則：
- **PRD 只寫 what/why**；功能分析歸 SA、頁面欄位 spec 歸 SRS（litmus：怎麼拆→SA、畫面怎麼操作→SRS）。
- R檔七區段、全域流水號 `R{4 位}`、PRD.md 純階段索引、traceability.md 由本 skill 生成。
- 來源未明確處列「待釐清」，**不腦補**。

## 來源檔判定
- **預設**讀 `PM/output/iniRFP.md`。
- 若使用者**指定了來源檔路徑**（如 `PM/Meeting/某次會議.md`），改讀該指定檔。
- 兩者皆無 → 提示使用者先跑「importRFP」產出 iniRFP.md，或指定來源檔，不繼續。

> 註：會議／訪談逐字稿屬自由格式，萃取較不結構化；若來源為 `PM/Meeting/` 之逐字稿，切分需求時更需逐筆與使用者確認。

## 步驟

### 1. 讀來源
依「來源檔判定」讀取全文。

### 2. 萃取需求項
辨識「獨立需求」，每個需求 = 一個 R檔。判定原則：
- 一個可獨立交付、可獨立驗收的業務目標 = 一個 R。
- iniRFP.md 多為**全域／非功能決策** → 這些通常不各自成 R，而是各 R檔「五、非功能需求」承接引用；只有明確的**功能型需求**才開 R檔。
- ⚠️ **不確定切分時，先列出建議的 R 清單（編號＋名稱＋階段＋切分理由）給使用者確認，不自行硬切。**

### 3. 配編號與階段
- 掃 `PRD/` 既有 R檔，取最大流水號後接續（全域唯一、不回收）。
- 階段：依使用者指示或來源標註填寫；不明則標「待確認」並於收尾提問。

### 4. 逐筆產出 R檔
- 複製 `PRD/R範本.md` → `PRD/R{編號}.md`。
- 填七區段。**功能分解、頁面行為、欄位一律不寫**（留 SA／SRS）。
- 非功能共通項引用 iniRFP.md，不重寫；只列該需求特有例外。
- 來源不足之處列「七、待釐清項目」。

### 5. 更新 PRD.md 階段索引
在對應階段加入條目：`- [[R{編號}] {名稱}](R{編號}.md)`。階段不存在則新增 H2 區段。

### 6. 生成 traceability.md
- **整份重新生成**：掃所有 R檔，輸出表格欄位 `R 編號 | 需求名稱 | 階段 | SA 對應 | SRS 對應 | 測試對應 | 狀態`。
- SA、SRS、測試 尚未撰寫 → 該格留 `—`（後續由 PRDtoSA／SAtoSRS／SRStoTest 回填）。
- 本檔由 skill 覆寫，提醒使用者勿手改。

### 7. 收尾提示
條列本次：新增／修改了哪些 R檔、PRD.md 異動、traceability.md 已重生；並彙整所有「待釐清項目」提醒使用者補確認。

## 原則
- **動工前先列「將產出哪些 R檔（編號＋名稱＋階段）」給使用者確認**，再逐筆寫（依 CLAUDE.md 通用守則）。
- PRD 只寫 what/why；功能分析歸 SA、頁面 spec 歸 SRS。
- 來源未明確處列「待釐清」，不腦補。
- 不主動執行 git。
