# SRStoTest（產測試案例）

把 SRS 的「畫面怎麼操作」與 PRD 的「驗收條件」轉成可執行的測試案例。一個 R 一個測試檔，TC 全域流水。

## 第 0 步：先讀規範
讀 `Testing/test-cases/rules/測試案例撰寫規範.md`（缺陷追蹤另見 `Testing/缺陷追蹤/rules/缺陷追蹤撰寫規範.md`）。鐵則：
- TC 從 **SRS 頁面**取操作步驟與畫面細節、從 **PRD 驗收條件**取「要驗什麼」。
- **15 欄固定表**；TC 編號全域唯一、不回收；每筆必填「需求編號」與「測試功能項目（SRS 代碼）」。
- 預期結果要**可判定通過與否**；SRS／PRD 未明處 → **標待釐清，不腦補預期結果**。
- **SRS／R檔有未解的「待釐清」→ 先請使用者澄清再產測試**；唯使用者明確堅持才帶著待釐清往下做（見步驟 1b）。

## 範圍判定
- 使用者**指定的 R／功能** → 只處理那些。
- **未指定** → 列已有 SRS 的功能（`SA/SA.md`／`SRS/`），請使用者選範圍。**建議一批對齊一個 sprint／階段**。

## 步驟

### 1. 讀來源與既有測試
讀目標 R檔（取「六、驗收條件」）、對應 SRS 頁面（步驟／欄位／驗證／訊息）、`SA/SA.md`（功能編號）；掃 `Testing/test-cases/` 既有檔取**最大 TC 號**續接。

### 1b. 盤點待釐清　★產測試前置閘
- 掃本批 SRS／R檔的「待釐清項目」，**逐項列給使用者確認／補答**。
- **未獲澄清前不產測試**；有答覆者回填來源，再往下。
- **例外**：使用者明確表示「先產、待釐清照標」時才繼續，並於對應 TC 的預期結果標「待釐清」。

### 2. 列 TC 規劃　★確認閘
產檔前先列下表，請使用者確認、修正後再寫：

| R | 對應 SRS 功能 | 預計場景數 | TC 起訖號 |
|---|---|---|---|

> 場景依規範 §六 CRUD 母版＋該頁 SRS／PRD 驗收條件展開；切分有疑義先問。

### 3. 逐 R 產測試案例檔
- 複製 `Testing/測試案例-範本.md` → `Testing/test-cases/{階段}/testing-cases-{階段碼}-{R編號}.md`。
- 套 CRUD 母版＋專屬場景，逐筆填 15 欄；TC 續號。
- 平台適用性（PC／Pad／Phone）依該系統目標載具填。

### 4. 回填 traceability.md「測試對應」欄
- 本批每個 R 的「測試對應」欄填測試案例檔連結（與 TC 區間）。
- 其他欄不動；此檔由 PRD 產出 skill 生成，本步只更新測試欄。

### 5. 缺陷追蹤（提示，不自動產）
- 測試**執行**後若發現問題，依 `Testing/缺陷追蹤/rules/` 用 `缺陷-範本.md` 建 Issue（檔名含對應 TC）。本 skill 不自動產 Issue。

### 6. 收尾提示
條列本批：新增哪些測試檔、TC 號區間、traceability 回填了哪些 R；彙整「待釐清項目」提醒使用者補確認。

## 原則
- **待釐清先澄清**：SRS／R檔有未解待釐清 → 先確認再產測試，除非使用者堅持（步驟 1b）。
- **先列 TC 規劃確認**，再逐筆寫。
- TC 從 SRS＋PRD 驗收條件來；預期結果可判定；未明處標待釐清，不腦補。
- TC 全域唯一、不回收；每筆回扣 R 與 SRS 代碼。
- 不主動執行 git。
