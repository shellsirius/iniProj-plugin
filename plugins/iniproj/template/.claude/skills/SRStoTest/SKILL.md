# 產測試案例（從 R檔）

以 **PRD R檔為起點**產測試案例：讀指定 R檔，順著追溯找到它對應的 SA 功能與 SRS 頁面，據以寫出可執行的測試案例。**一個 R 一個測試檔**，TC 全域流水。

> **缺陷 Issue 是另一件事**：測試執行後記錄問題的流程，見 `Testing/缺陷追蹤/`，**不屬本 skill**。

## 第 0 步：先讀規範
讀 `Testing/test-cases/rules/測試案例撰寫規範.md`。鐵則：
- **以 R檔錨定**：每個測試檔對應一個 R；**先讀整份 R檔**——§三 使用情境（要驗哪些情境）、**§四 業務規則與限制（反向／邊界／卡控案例來源）**、§六 驗收條件（驗收點）；§五 特有非功能例外視需要、§七 待釐清進前置閘——再從**追溯到的 SRS 頁面**取操作步驟與畫面細節。
- **15 欄固定表**；TC 全域唯一、不回收；每筆必填「需求編號」與「測試功能項目（SRS 代碼）」。
- 預期結果要**可判定通過與否**；SRS／PRD 未明處 → **標待釐清，不腦補**。
- **R檔／SRS 有未解的「待釐清」→ 先請使用者澄清再產測試**；唯使用者明確堅持才帶著待釐清往下做（見步驟 1b）。

## 範圍判定
- 使用者**指定的 R檔**（如「為 R0001 產測試」「R0001–R0003」）→ 只處理那些。
- **未指定** → 列 `PRD/PRD.md`／`PRD/` 既有 R檔，請使用者選。**建議一批對齊一個 sprint／階段**。

## 步驟

### 1. 讀 R檔並追溯 SA／SRS
- **讀整份指定 R檔**，掌握：§二 需求說明、§三 使用情境（情境）、**§四 業務規則與限制（反向／邊界／卡控案例來源）**、§六 驗收條件（驗收點）；§五 特有非功能例外視需要納入、§七 待釐清進步驟 1b。
- 依 `PRD/traceability.md`（必要時 `SA/SA.md`）**追溯該 R 對應的 SA 功能與 SRS 頁面**；讀那些 SRS 頁面取操作步驟／欄位／驗證／訊息細節。
- 以 R檔的**情境＋業務規則＋驗收條件**為「要驗什麼」，SA／SRS 為「怎麼操作」，兩者合起來寫案例（正向＋反向／邊界）。
- 掃 `Testing/test-cases/` 既有檔取**最大 TC 號**續接。
- 若該 R 追溯不到 SRS（traceability 為 —）→ 提示使用者該 R 宜先產 SRS，或僅依 R 驗收條件寫概略案例（先問，不硬補）。

### 1b. 盤點待釐清　★產測試前置閘
- 掃該 R檔與其 SRS 的「待釐清項目」，逐項列給使用者確認／補答。
- **未獲澄清前不產測試**；例外：使用者明確「先產、待釐清照標」才續，預期結果處標待釐清。

### 2. 列 TC 規劃　★確認閘
產檔前先列下表，請使用者確認、修正後再寫：

| R | 對應 SRS 功能 | 預計場景數 | TC 起訖號 |
|---|---|---|---|

> 場景依規範 §六 CRUD 母版＋該頁 SRS／R 驗收條件展開；切分有疑義先問。

### 3. 逐 R 產測試案例檔
- 複製 `Testing/測試案例-範本.md` → `Testing/test-cases/{階段}/testing-cases-{階段碼}-{R編號}.md`（階段＝該 R 的階段）。
- 套 CRUD 母版＋專屬場景，逐筆填 15 欄；TC 續號；平台適用性（PC／Pad／Phone）依系統載具。

### 4. 回填 traceability.md「測試對應」欄
- 本批每個 R 的「測試對應」欄填測試檔連結（與 TC 區間）；其他欄不動。

### 5. 收尾提示
條列本批：新增哪些測試檔、TC 號區間、traceability 回填了哪些 R；彙整「待釐清項目」提醒使用者補確認。

## 原則
- **從 R檔產測試**：先讀整份 R檔（情境＋業務規則＋驗收條件），再依追溯找 SA／SRS 取操作細節；一個 R 一個測試檔。
- **待釐清先澄清**（步驟 1b）；先列 TC 規劃確認再寫。
- TC 全域唯一、不回收；預期結果可判定，未明處標待釐清、不腦補。
- **缺陷 Issue 是獨立流程**，不在本 skill；測試執行後依 `Testing/缺陷追蹤/` 另記。
- 不主動執行 git。
