# MeetingtoPRD

把會議／訪談逐字稿（自由格式）萃成 PRD 需求檔。**與 `iniRFPtoPRD` 同產出、同規範**；差別在來源是非結構化逐字稿——需求要靠判斷力抽，且常是**修訂既有需求**而非全新。

## 第 0 步：先讀規範
讀 `PRD/rules/PRD撰寫規範.md`，全程依其格式。三鐵則同 iniRFPtoPRD：
- **PRD 只寫 what/why**；功能分析歸 SA、頁面 spec 歸 SRS（litmus：怎麼拆→SA、畫面怎麼操作→SRS）。
- R檔七區段、全域流水號 `R{4 位}`、PRD.md 純階段索引、traceability.md 由 skill 生成。
- 來源未明確處列「待釐清」，**不腦補**。

## 來源檔判定
- 使用者**指定了檔案路徑**（如 `PM/Meeting/2026-06-20訪談.md`）→ 讀該檔。
- **未指定** → 列出 `PM/Meeting/` 內檔案請使用者選一份。**一次只處理一份會議記錄**，不把多份混在一起萃。
- 讀檔格式：`.docx`（macOS `textutil -convert txt -stdout`／Windows PowerShell 解 zip 取 `word/document.xml`）、`.pdf`（pdftotext／Read）、`.txt`／`.md` 直接讀。

## 步驟

### 1. 讀來源、抓會議基本資訊
讀全文。抓「會議日期／場次／與會者」（標題或內文）；抓不到日期 → 問使用者。日期用於 R檔「來源」標註。

### 2. 萃取需求候選（本 skill 重點）
逐字稿混著需求、討論、雜訊，辨識時：
- **抓**：明確的需求與決議（客戶要什麼、定案的業務規則）。
- **排除**：尚在討論未定案、被否決的方案、純背景、與需求無關的行政事項。
- 每個候選需求**附逐字稿佐證**（引用原句或段落位置），供使用者核對。
- 高度判準同規範：PRD 只留 what/why；怎麼拆→SA、畫面怎麼操作→SRS。
- 「討論到但沒定案」→ 不直接成需求，列為對應 R 的「待釐清項目」。

### 3. 比對既有 PRD：新增 or 修訂？　★與 iniRFPtoPRD 最大不同
會議常是「改既有需求」而非全新。掃 `PRD/` 既有 R檔，逐筆判定：
- 對既有需求的補充／變更 → **修訂該 R檔**（直接改寫對應區段；不在文件內記變更紀錄，版本演進由 git 追蹤）。
- 全新需求 → 取下一個流水號開新 R檔。
- 不確定對應到哪個既有 R → **問使用者，不亂歸**。

### 4. 列清單給使用者確認　★動工前必做
產檔／改檔前，先列下表請使用者確認、修正切分後再寫：

| 候選 | 新增 / 修訂(R 編號) | 一句話需求 | 逐字稿佐證 |
|---|---|---|---|

### 5. 逐筆產出／修訂 R檔
- **新增**：複製 `PRD/R範本.md` → `PRD/R{編號}.md`。
- **修訂**：在既有 R檔對應區段直接改寫（不加變更紀錄）。
- 「來源」標註該會議（如 `PM/Meeting/2026-06-20訪談.md` 或 `115/06/20 會議紀錄`）。
- **功能分解、頁面行為、欄位不寫**（留 SA／SRS）；非功能共通項引用 iniRFP.md；缺漏列「待釐清項目」。

### 6. 更新 PRD.md 階段索引
- 新增需求 → 於對應階段加入條目。
- 修訂既有需求 → 索引不動（除非階段改變）。

### 7. 生成 traceability.md
整份重新生成：掃所有 R檔，輸出 `R 編號 | 需求名稱 | 階段 | SA 對應 | SRS 對應 | 測試對應 | 狀態`；下游未寫處留 `—`。本檔由 skill 覆寫，提醒使用者勿手改。

### 8. 收尾提示
條列本次：新增了哪些 R、修訂了哪些 R（各補了什麼）、PRD.md／traceability.md 異動，並彙整所有「待釐清項目」提醒使用者補確認。

## 原則
- **一次處理一份**會議記錄。
- **動工前先列需求清單**（含逐字稿佐證、新增/修訂判定）給使用者確認，再寫。
- 非結構化來源更要保守：寧可列「待釐清」、問使用者，**不腦補、不硬切**。
- PRD 只寫 what/why；功能分析歸 SA、頁面 spec 歸 SRS。
- 不主動執行 git。
