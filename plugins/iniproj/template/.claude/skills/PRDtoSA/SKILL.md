# PRDtoSA（流程先行）

把 PRD 的「要什麼」轉成 SA 的「功能架構」。順序：**需求分類 → 理流程線 → 沿流程拆功能 → 模組 SA → 雙軸追溯**。產出的「功能」是未來一個 SRS 頁面。

## 第 0 步：先讀規範
讀 `SA/rules/SA撰寫規範.md`（並順帶看 `PRD/rules/PRD撰寫規範.md` 了解 R檔結構）。鐵則：
- **流程先行**：有業務流程的需求，先理流程線再拆功能；不要急著拆。
- **三型分類**：流程型 → 流程線→拆功能；扁平功能型 → 直接拆功能；**純非功能型 → 不下 SA**（不拆功能、不進流程/模組）。
- **R↔流程 M:N**：接合點是流程步驟，不是整個 R；路由先確認，不假設。
- SA 只寫怎麼拆／相依／流程／資料；不寫業務動機（PRD）、頁面欄位／元件（SRS）。§2 只寫跨功能／跨模組相依。
- 功能切分／模組歸屬／系統代碼／流程切分不明確 → **問使用者，不腦補**。

## 範圍判定
- 使用者**指定的 R 批次或模組** → 只處理那些。
- **未指定** → 列 `PRD/PRD.md` 索引或 `PRD/` 既有 R檔，請使用者選範圍。**建議一批對齊一個 sprint**。

## 步驟

### 1. 讀 PRD 與既有 SA
讀目標 R檔；並讀 `SA/SA.md`（流程線索引＋功能索引）、`SA/flows/` 既有流程線、`SA/{系統}/` 既有模組，掌握現有流程線、模組劃分與已用編號（避免重號、續接流程步驟 ID）。

### 2. 分類 → 路由　★第一道閘
- 把本批**每個 R 分三型**（流程型／扁平功能型／純非功能型；邊界見規範 §三）。
- **流程型**：路由到流程線（一個 R 可跨多條流程線）；定位既有 `SA/flows/F*` 或新建；把本批步驟接進流程線，**未到的步驟標 ⬜，不捏造**；步驟用穩定 ID `F{n}-S{序}`（插入用 S2.5 或續號，不回收）。
- **扁平功能型**：不建流程線，直接進步驟 3。
- **純非功能型**：列入「本批不下 SA 清單」（步驟 7 於 traceability 標記）。
- **把下列三者列給使用者確認**，有疑義先問：
  1. 每個 R 的分類；2. 流程型 R→流程線的路由（尤其一 R 跨多流程怎麼切）＋流程線這一段步驟；3. 本批不下 SA 的清單。

### 3. 沿流程拆功能、定模組、配編號
- 只針對**本批涵蓋的步驟／扁平型需求**拆功能（每步驟 → 一或多功能 ＝ 未來一個 SRS 頁面）。
- 歸入**模組**；若需新模組，定模組編號與**系統代碼**（沿用既有；首次或不確定 → 問使用者）。
- 配**功能編號** `{系統}-{模組}-{功能}`（接續既有、不回收；單系統專案可省系統層）。

### 4. 列功能／模組規劃表　★第二道閘
產檔前先列下表，請使用者確認、修正後再寫：

| 功能編號 | 功能名稱 | 流程步驟 | 模組 | 來源 R | 新增模組／沿用既有 |
|---|---|---|---|---|---|

> 切分、模組歸屬、系統代碼有疑義一律先問。

### 5. 寫／更新模組層 SA
- **新模組**：複製 `SA/模組-範本.md` → `SA/{系統}/{系統}-{模組}.md`，填 §1–§5。
- **既有模組**：把新功能併入 §2 功能清單，視情況更新其餘。
- §1 流程歸屬：指向流程線與步驟（跨模組流程以流程線為權威，**不重寫**）；扁平型寫「無多步流程」。
- §2 功能清單含「流程步驟」欄（扁平型填 —）。§3 只寫跨功能／跨模組相依（界線見規範 §二、§七）。
- 不寫頁面欄位／元件（SRS）、不寫業務動機（PRD）；PRD 未明處列「待釐清」。

### 6. 更新流程線檔與 SA.md
- 流程線檔：本批落地的步驟狀態改 ✅、掛上對應功能編號與來源 R。
- `SA.md`：功能架構索引逐列登錄（系統｜模組｜功能編號｜功能名稱｜來源需求）；更新流程線索引的涵蓋狀態。

### 7. 回填 traceability.md「SA 對應」欄（需求視角對帳）
- 本批每個 R（含被拆到多流程者）「SA 對應」欄填對應**模組 SA／功能**連結。
- **純非功能型 R** → 標「**非功能，不下 SA（見 PRD 全域約束）**」，不留白。
- 其他欄位不動；此檔由 PRD 產出 skill 生成，本步只更新 SA 欄。

### 8. 收尾提示
條列本批：新增／更新了哪些流程線、模組 SA、SA.md 新增哪些功能、traceability 回填了哪些 R；**明列本批涵蓋了哪些流程步驟、哪些仍 ⬜、哪些 R 判為不下 SA**；彙整「待釐清項目」提醒使用者補確認。

## 原則
- **流程先行**：有流程才先理流程線；扁平型直接拆；純非功能型不下 SA。
- **兩道閘**：先確認分類＋流程線路由，再確認功能／模組規劃，才動手寫。
- **M:N 路由先確認**，不假設一 R 對一流程。
- 流程步驟**穩定 ID**、不回收不重排；功能編號全域唯一。
- **雙軸追溯**：流程線（流程視角）＋ traceability（需求視角）＋ SA.md（功能地圖）。
- 功能切分／模組歸屬／系統代碼／流程切分不確定就問，**不腦補**。
- 建議**一批（sprint）對齊處理**。
- 不主動執行 git。
